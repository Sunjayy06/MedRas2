════════════════════════════════════════════════════════

Add these endpoints alongside the existing ones:

POST /api/stats/convert-format
  Input:  Excel file + format declaration (wide/long)
          + patient ID column name
          + timepoint column names (list)
          + variable columns to melt (list)
  Output: Converted long-format Excel file for download
          + preview of first 10 rows
  Logic:  pandas.melt() for wide-to-long
          Validate: same patient appears correctly per row

POST /api/stats/generate-dummy
  Input:  study_type (anaemia/diabetes/hypertension/custom)
          n_patients (integer)
          n_groups (1/2/3)
          custom_variables (list, only if custom)
          missing_pct (default 5%)
  Output: Generated Excel file for download
          + dataset summary (n rows, columns, distributions)
  Logic:  numpy random with realistic medical distributions
          Log-normal for lab values (always positive)
          Normal for age, weight, height
          Binomial for categorical variables

POST /api/stats/power-check
  Input:  n_collected (integer)
          study_type (string)
          alpha (default 0.05)
          effect_size (detected from data or user input)
          n_groups (integer)
  Output: achieved_power (%)
          verdict: "Adequate (≥80%)" / 
                   "Borderline (60-79%)" / 
                   "Underpowered (<60%)"
          recommendation: continue / stop / add patients
  Logic:  statsmodels.stats.power for power calculation
          pingouin.power_ttest / power_anova / power_chi2

POST /api/stats/merge-datasets
  Input:  List of Excel files (from multiple users/doctors)
          merge_key: column name to match on 
                     (Patient_ID / Phone / Aadhaar / Family_ID)
          conflict_resolution: keep_first / keep_latest / flag
  Output: Merged Excel dataset
          Conflict report: rows that could not be matched
          Duplicate report: exact duplicates removed
  Logic:
    Step 1: Load all files into separate DataFrames
    Step 2: Attempt merge on primary key (Patient_ID)
    Step 3: If no match, try secondary keys 
            (Phone, then Aadhaar, then Family_ID)
    Step 4: For duplicates with different values:
              If conflict_resolution = keep_latest:
                Use row with most recent date/timestamp
              If conflict_resolution = flag:
                Keep all, add "CONFLICT" column = True
    Step 5: Report: n merged, n conflicts, n duplicates removed

GET /api/stats/progress/{job_id}
  Returns: current phase name, % complete, 
           estimated time remaining
  Use Server-Sent Events (SSE) for real-time updates

    10% — Reading Excel file

    40% — Running normality tests
    50% — Executing statistical tests
    65% — Generating graphs
    80% — Writing interpretation

ADDENDUM C — SAMPLE SIZE INTEGRATION
════════════════════════════════════════════════════════

After Phase 1 (Excel upload), before Phase 5 
(variable assignment), add this integration step:

STEP: SAMPLE SIZE VALIDATION

  C1: ASK USER
    "Do you know the required sample size for your study?"
    Options:
      A) Yes — enter it: [number input]
      B) Calculate it now (link to sample size module)
      C) Skip (analyse with available data)

  C2: IF SAMPLE SIZE KNOWN OR CALCULATED
    Compare: n_required vs n_collected
    
    If n_collected >= n_required:
      Show GREEN banner:
      "Your dataset has [n] patients. 
       Required sample size: [n_required]. 
       Your study is adequately powered."
    
    If n_collected is 80-99% of n_required:
      Show AMBER banner:
      "Your dataset has [n] patients. 
       Required sample size: [n_required]. 
       You are [X] patients short. 
       Study may be slightly underpowered. 
       Proceed with caution."
    
    If n_collected < 80% of n_required:
      Show RED banner:
      "WARNING: Your dataset has [n] patients 
       but [n_required] are required. 
       This study is underpowered — results 
       may miss real effects (Type II error risk). 
       Options:
       [Collect more data] [Proceed anyway with disclaimer]"
    
    If user proceeds with underpowered data:
      Add automatic disclaimer to report:
      "LIMITATION: The sample size in this study 
      ([n] participants) is below the calculated 
      requirement ([n_required]). Results should 
      be interpreted with caution. The study may 
      be underpowered to detect the anticipated 
      effect size, and findings should be validated 
      in a larger cohort."

  C3: POST-HOC POWER CALCULATION
    After running any comparison test, 
    automatically calculate and display:
    
    "Achieved statistical power: [X]%"
    
    For t-test:
      statsmodels.stats.power.TTestIndPower
      solve_power(effect_size=cohen_d, 
                  nobs1=n_group1, alpha=0.05, 
                  ratio=n_group2/n_group1)
    
    For ANOVA:
      statsmodels.stats.power.FTestAnovaPower
    
    For Chi-square:
      statsmodels.stats.power.GofChisquarePower
    
    Show in results panel:
    "Power of this test: [X]%
     [Interpretation: adequate ≥80% / 
                      borderline 60-79% / 
                      underpowered <60%]"
    
    Include in report footnote for each table.

ADDENDUM G — CROSS-VARIABLE CLASSIFICATION
════════════════════════════════════════════════════════

Handle variables that behave differently by context:

RULE 1 — Age context detection:
  If objective contains "age group" / "age category":
    Treat Age as ORDINAL (grouped)
    Auto-create groups: <30, 30-45, 46-60, >60
  If objective contains "age" as predictor in regression:
    Treat Age as SCALE (continuous)
  Default: treat as SCALE, offer grouping option

RULE 2 — Score variables:
  Columns named: score, index, scale, VAS, NRS, GCS, MMSE
  Default to ORDINAL
  BUT if used as outcome in regression: treat as SCALE
  Show: "Pain score treated as ordinal for descriptives
         and as continuous in regression. Correct?"

RULE 3 — Binary variables in regression:
  0/1 binary in descriptives: NOMINAL (frequency + %)
  Same 0/1 as DV in regression: binary outcome → logistic
  Same 0/1 as IV in regression: dummy variable → linear
  System handles this automatically, no user action needed

RULE 4 — Percentage variables:
  Values between 0-100 that are proportions (SpO2, EF):
    Treat as SCALE for descriptives
    Check: if used as DV and binary outcome possible,
    offer to dichotomise (e.g. EF < 40% vs ≥ 40%)

RULE 5 — Likert scale variables (1-5 or 1-7):
  Default: ORDINAL
  If 5+ items from same questionnaire detected:
    Offer to compute composite score:
    "We detected [n] Likert-scale items that 
     may form a composite score. 
     Compute total/average score?" Yes/No
  If Yes: treat composite as SCALE
