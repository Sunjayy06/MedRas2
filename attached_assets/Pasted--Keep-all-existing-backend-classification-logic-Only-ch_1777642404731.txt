CHANGE 4 — ISSUE DETECTION LOGIC
════════════════════════════════════

Before rendering Step 3, the backend should 
run these checks on every column and return 
an issues list:

  Issue type: "text_in_numeric"
    When: column classified as Nominal but 
    values look like "Grade 4", "Level 3", 
    "Stage 2" — number with a text prefix.
    Flag: show amber sub-line below column name 
    in the table.
    Assistant pre-message: mention this column 
    and offer to strip the prefix.

  Issue type: "low_unique_nominal"
    When: column classified as Ordinal but 
    has only 2 unique values.
    Flag: amber sub-line 
    "Only 2 unique values — may be Nominal (Yes/No)"

  Issue type: "high_missing"
    When: > 30% missing.
    Flag: amber missing count in table.
    No assistant message needed — covered in Step 5.

  Issue type: "duplicate_values"
    When: Nominal column has the same label 
    repeated with different capitalisation 
    (NECK FRACTURE vs Neck Fracture vs neck fracture).
    Flag: amber sub-line 
    "Inconsistent casing detected — may be the same value"

  Issue type: "numeric_as_id"
    When: column classified as ID but values 
    are small integers that look like a scale.
    Flag: amber sub-line 
    "Classified as ID — verify this is not a score"

The issues list is passed to the frontend.
The "Issues flagged" summary card shows 
the count of columns with any issue.
The assistant's initial message mentions 
any detected issues by name.
